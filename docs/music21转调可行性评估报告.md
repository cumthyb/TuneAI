# music21 转调可行性评估报告（TuneAI）

## 1. 背景与目标

TuneAI 当前目标是将简谱图片识别为结构化事件，并输出指定目标调的简谱结果。  
本报告评估：在现有后端架构中引入 `music21` 做转调与乐理校验，是否可行、如何低风险落地、以及需要关注的性能与兼容性边界。

## 2. 现状梳理（基于当前仓库）

- 依赖层面：`music21` 已在 `pyproject.toml` 与 `poetry.lock` 中存在，无需新增包管理改造。
- 转调入口：主流程在 `backend/tuneai/core/application/pipeline.py` 的 `[4/7] transpose` 步骤。
- 转调实现：核心在 `backend/tuneai/core/domain/music.py` 的 `transpose_score_ir(...)`。
- 数据模型：`backend/tuneai/schemas/score_ir.py` 采用扁平 `events`（`NoteEvent`/`RestEvent`），暂无小节、时值、和弦、局部转调事件等结构。
- 热路径特征：同一请求中包含 OCR、VL/LLM 校验、图像渲染，当前已是多段 CPU + 网络混合链路。

## 3. 可行性结论

结论：**可行，且接入门槛低；建议分阶段落地。**

原因：

1. 依赖已经就绪，接入成本主要在代码语义对齐，而非环境搭建。
2. 现有转调逻辑有明确挂载点，可通过策略开关实现“可灰度、可回退”。
3. `music21` 能补足等音拼写、调性上下文相关的表达能力，为后续复杂场景（临时升降、和弦、局部调性变化）提供扩展空间。

## 4. 主要风险与影响

### 4.1 语义对齐风险（最高）

当前 `ScoreIR` 较轻量，和 `music21` 的完整乐谱模型存在抽象层级差异：

- 目前主要是“度数 + 升降记号 + 八度偏移”的事件，不含时值/小节上下文。
- 直接引入完整乐理推导时，可能出现与当前输出行为不一致。

**建议**：先并跑对比，建立“可接受差异清单”，避免一次性替换。

### 4.2 性能与时延风险

`music21` 不是轻量库，完整分析可能增加请求耗时。  
在当前 API 热路径内，若同步执行复杂转调/分析，可能放大尾延迟。

**建议**：

- 使用懒加载（仅在需要 `music21` 路径时 import）。
- 在异步流水线中用线程执行 CPU 计算（避免阻塞事件循环）。
- 先灰度上线并记录耗时与差异样本。

### 4.3 回归风险

现有测试期望基于 legacy 逻辑，若直接替换实现，容易导致大面积断言失效。

**建议**：新增双实现对照测试（legacy vs music21）与回归样本集，再逐步切换默认策略。

## 5. 推荐落地方案（低风险）

### 阶段 1：并行 PoC（不改变对外行为）

- 在 `backend/tuneai/core/domain/music.py` 增加 `music21` 转调实现函数。
- 保持 `legacy` 为唯一对外输出，`music21` 仅用于内部对比与日志抽样。
- 输出 PoC 报告：一致率、差异类型、差异样本。

### 阶段 2：策略开关灰度

- 在 `backend/tuneai/core/application/pipeline.py` 的转调步骤加入策略开关：
  - `legacy`（默认）
  - `music21`
- 支持快速回切，确保线上安全。

### 阶段 3：按需扩展数据模型

- 若要使用 `music21` 的更深能力，再扩展 `ScoreIR`（如和弦/局部转调事件、时值信息）。
- 保证渲染链路能从新模型回写到当前可绘制表达。

## 6. 验收标准

- 正确性：目标调音高映射正确，升降号拼写符合目标调惯例。
- 兼容性：`/api/transpose` 的输入输出协议不变。
- 性能：开启 `music21` 后新增耗时在可接受范围内（需有实测）。
- 可回退性：可在配置层一键回退到 `legacy`。

## 7. 建议监控指标

- 转调阶段耗时（P50/P95/P99）
- 请求总耗时与超时率
- legacy 与 music21 差异率（按调号/音符类型分桶）
- 关键告警：异常率、回退次数、失败样本数

## 8. 最终建议

在 TuneAI 当前架构下，`music21` 是一个**值得引入但应分阶段控制风险**的方案。  
短期建议优先完成“并行 PoC + 开关灰度 + 对照测试”，在不影响现网行为的前提下获取真实收益数据，再决定是否扩大到默认路径与模型层扩展。
