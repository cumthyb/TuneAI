# TuneAI 完整执行方案

## 1. 方案定位

本方案基于以下已确认约束重新设计：

- 仅处理印刷版简谱，不支持手写简谱
- `0` 视为休止符，需要识别，但不参与移调
- 输出图片尽量保留原版布局，不做整体重新排版
- 不引入训练型机器学习模型
- 识别主链路采用 `OCR + OpenCV 几何规则 + LangChain 驱动的 LLM 纠错/补全`
- 由单个开发者在 2 到 3 个月内完成可用原型
- **前后端均需建立完整日志系统，便于调试与问题排查**

这意味着 TuneAI 当前方案的核心难点不再是“训练符号检测器”，而是：

1. 如何把简谱拆成稳定的可解析符号单元
2. 如何把数字、升降号、八度点、时值符号正确绑定到同一个音符事件
3. 如何在不重排整页的前提下，将变化后的内容贴回原图
4. 如何把每个音符解码为十二平均律音高后再正确编码回目标调简谱

## 2. 目标与非目标

### 2.1 目标

- 输入一张简谱图片（PNG/JPG，不支持 PDF）
- 识别全局调号，如 `1=C`、`1=G`、`1=Eb`
- 识别简谱主体中的音符与结构元素
- 识别显式转调标记，如行内再次出现的 `1=D`
- 将乐谱转换到用户指定目标调
- 输出保留原布局风格的新图片

### 2.2 支持范围

- 音符：`1 2 3 4 5 6 7`
- 休止符：`0`
- 升降号：`#`、`b`
- 八度点：上点、下点
- 常见时值符号：`-`、`_`
- 倚音
- 小节线
- 调号表示：`1=C` 一类文本
- 显式转调

### 2.3 非目标

- 五线谱支持
- PDF 输入（仅支持 PNG/JPG 图片）
- 手写简谱支持
- 歌词语义理解
- 和弦进行分析
- 隐式转调推断
- 复杂出版级重排

说明：歌词和和弦文字在当前方案中按“背景文字”处理，不做语义解析，也不主动修改，除非它们与调号文本 `1=X` 混淆。

## 3. 核心判断

### 3.1 关于移调本身

本项目中的“移调”必须定义为基于十二平均律的逐音符音高变换，而不是只把 `1=` 文本改掉。

具体来说，后端必须对每一个音符事件执行以下过程：

1. 根据当前调号上下文，将 `degree + accidental + octave_shift` 解码为十二平均律下的绝对音高
2. 根据原调和目标调之间的移调间隔，对该音高执行半音级平移
3. 将平移后的音高重新编码为目标调上下文下的简谱表示
4. 将新的数字、升降号、八度点重新写回原图

这意味着系统的真实主路径应定义为：

`识别简谱 -> 解析当前调上下文 -> 逐音符解码到 12-TET -> 按半音移调 -> 重新编码简谱 -> 原位覆盖输出`

而不是：

`只替换页首 1=`

需要特别说明的是：

- 在某些标准相对简谱片段中，经过严格的 12-TET 解码与重编码后，输出数字可能碰巧与原谱一致
- 但工程实现上不能依赖这个现象
- 否则系统无法正确处理局部转调、临时升降、等音拼写和后续扩展

例如：

- 原谱：`1=G`，音符 `5`
- 先解码为绝对音高 `D`
- 若目标调为 `C`，则先计算移调间隔，再把 `D` 平移到目标音高
- 最后在 `1=C` 语境下重新编码该音符

即便最终结果仍然写成 `5`，它也必须是“重编码结果”，而不是文本层面的快捷替换。

### 3.2 关于保留原排版

保留原排版在本项目里是可行的，但要明确边界：

- 优先做“原位贴片替换”，不是版面重流
- 替换单位应是“音符簇”，即主数字、升降号、八度点的组合
- 时值符号通常保持不变，但其附着对象变化后仍需重新校验位置
- 不能预设只有调号区域会变化，逐音符移调后可能需要改写整行中的多个音符簇
- 若替换内容宽度明显变大，允许在局部范围内缩放或触发人工确认标记

这条路径比“全页重新排版”更适合当前方案。

## 4. 高层架构设计

本项目应明确拆成前端和后端两个子系统，避免后续把交互逻辑、任务状态和识别流水线耦合在一起。

### 4.1 前端职责

前端建议做成一个非常轻的 Web 页面层，不单独引入 React 这类 SPA 框架。当前方案更适合由后端直接提供模板页面和静态资源。前端只负责：

1. 上传简谱图片（PNG/JPG）
2. 选择目标调
3. 展示处理中状态（同步等待时的 loading）
4. 展示原图与结果图对比
5. 展示低置信度告警与人工复核提示
6. 下载结果图和解析 JSON
7. **前端日志**：记录用户操作、请求/响应摘要、错误信息，支持控制台输出与可选远程上报，便于复现问题

前端不承担任何 OCR、移调和图像处理逻辑，只负责输入、展示和交互。

### 4.2 后端职责

后端负责全部核心处理：

1. 图片接收与临时存储
2. 图像预处理
3. 行/小节切分
4. OCR 与 OpenCV 规则识别
5. 符号绑定与语义解析
6. 十二平均律音高解码、逐音符移调与局部转调处理
7. 原位渲染与结果导出
8. 结果返回
9. **后端日志**：全链路结构化日志，覆盖各阶段耗时、中间结果摘要、错误堆栈，支持按请求 ID 追踪，便于调试

### 4.3 推荐交互方式（简化版）

为快速落地，初版采用**单张图片同步处理**模式：

1. 前端上传图片并提交目标调
2. 后端同步执行完整处理流程（目标 5-15 秒）
3. 后端直接返回结果图、解析 JSON、warning 列表

简化理由：
- 单用户、单张图片场景，无需任务队列
- 避免任务状态管理的额外复杂度
- 印刷简谱识别通常 5-15 秒可完成

超时处理：若处理超过 30 秒，返回超时错误，提示用户重试或检查图片质量。

### 4.4 系统主链路

简化流程：

```
用户上传图片 -> 后端同步处理 -> 返回结果图+JSON
```

后端内部流水线：

```
image -> preprocess -> segment -> OCR/CV -> token graph -> score IR -> transpose -> overlay render -> output
```

### 4.5 技术选型建议

为控制单人开发复杂度，建议：

- 前端：`Jinja2 Templates + Vanilla JS`
- 后端：`FastAPI + Python`，**同步处理接口**
- LLM 编排：`LangChain`（纠错、低置信度补全）
- 文件存储：临时目录，不引入数据库
- 无任务队列、无并发处理、无用户系统（单张图片场景）

## 5. 详细处理流程

### 5.1 输入与预处理

输入：

- 一张 PNG 或 JPG 格式的简谱图片（**不支持 PDF**，用户需自行将 PDF 转为图片后上传）
- 用户指定目标调，如 `C`、`G`、`Eb`

预处理步骤：

1. 灰度化
2. 去噪
3. 二值化
4. 倾斜校正
5. 对比度增强
6. 页面边缘裁切

目标：让数字、横线、八度点、调号文本尽可能清晰。

### 5.2 版面切分

使用 OpenCV 的投影分析与连通域方法做两类切分：

1. 行切分：找出每一行简谱的垂直范围
2. 小节辅助定位：识别小节线，构建行内分段

输出：

- 每一行的边界框
- 每一小节的大致边界
- 每个候选符号的局部图像区域

### 5.3 OCR 与几何识别

这里不做训练模型，采用混合策略：

#### A. OCR 负责识别的内容

- 音符数字：`0-7`
- 文本调号：`1=C`、`1=Eb`
- 行内再次出现的 `1=X` 转调标记

#### B. OpenCV 几何规则负责识别的内容

- 八度点
- 时值横线 `-`
- 下划线 `_`
- 小节线
- 倚音与主音的相对位置关系
- 升降号与数字的邻接关系

#### C. LangChain + LLM 负责的内容

- 用 LangChain 封装提示词、模型调用和结构化输出
- 对 OCR 输出的候选进行结构纠错
- 对低置信度小节做二次解释
- 在“多个符号绑定方案”之间做选择
- 统一处理重试、fallback 和输出校验

注意：

- LangChain 负责流程编排，不负责替代核心规则逻辑
- LLM 不直接承担整页视觉识别
- LLM 只处理已经过 OCR/CV 提取的中间结构
- 这样可降低幻觉风险并提升可调试性

### 5.4 符号关系绑定

这是当前方案最关键的步骤。

要把下列离散检测结果绑定成单个音符事件：

- 主数字
- 数字左侧的升降号
- 数字上方或下方的八度点
- 数字右侧或下方的时值符号
- 前置倚音组

建议规则：

1. 先按行建立阅读顺序
2. 在每行内按 `x` 坐标排序主数字
3. 在每个主数字周围建立局部窗口
4. 将窗口内的附属符号按相对位置绑定到该数字
5. 若一个符号同时可绑定多个数字，按距离、方向和类型优先级评分

输出结果不是纯文本，而是结构化事件流。

### 5.5 乐谱语义解析

解析器将 token 序列转换成内部乐谱表示，内容包括：

- 全局调号
- 小节列表
- 每小节的音符事件
- 转调事件
- 倚音组
- 休止符事件

同时做最基本的一致性检查：

- 调号格式是否合法
- 同一位置是否出现冲突符号
- 小节内是否存在明显无法解释的残余符号

### 5.6 移调处理

移调逻辑如下：

1. 识别原始全局调号 `source_key`
2. 读取用户指定目标调 `target_key`
3. 为每个音符建立生效中的调号上下文，包含全局调号和显式局部转调
4. 将每个音符解码为十二平均律下的绝对音高
5. 计算原调到目标调的移调间隔 `delta`
6. 对每个音符的绝对音高执行 `+delta` 或 `-delta` 的半音平移
7. 将平移后的音高在目标调上下文下重新编码为 `degree + accidental + octave_shift`
8. 将所有显式局部转调标记同步移调并重新编码
9. 保持休止符 `0` 不变

也就是说，当前方案必须执行逐音符的绝对音高重写，只是最终输出仍然是简谱格式。

### 5.7 原位渲染与贴回

这是输出层的核心。

建议采用“局部擦除 + 局部重绘 + 贴回原图”的方案：

1. 找到所有变化的 token 区域
2. 对原 token 边界框做少量扩边
3. 用局部背景估计或图像修补清除旧符号
4. 在透明画布上绘制新符号
5. 按原基线和字号估计结果贴回图中
6. 导出新图片

要点：

- 非变化区域保持原图不动
- 变化区域以最小影响原则处理
- 对调号文本这类多字符内容，优先使用整块替换
- 对音符区域，应以“主数字 + 升降号 + 八度点”为一个最小渲染单元
- 时值线、下划线等节奏信息原则上继承原位置，但在主数字宽度变化后需要重新对齐

### 5.8 结果校验

输出前做三类检查：

1. 结构检查：是否成功识别到全局调号
2. 覆盖检查：所有替换区域是否已成功写回
3. 置信度检查：低置信度处是否需要提示人工复核

## 6. 主要系统模块

模块设计也应明确区分前端模块与后端模块。

### 6.1 前端模块

前端保持极简，不使用 React。建议以模板页面加少量原生 JavaScript 完成。

#### 6.1.1 upload

职责：

- 选择或拖拽上传图片
- 输入目标调
- 发起处理任务

#### 6.1.2 loading-state

职责：

- 同步模式下展示「处理中」loading 状态（无轮询）
- 展示失败原因与错误提示

#### 6.1.3 result-viewer

职责：

- 展示原图与结果图
- 高亮变化区域
- 展示 warning 和低置信度区域

#### 6.1.4 download-panel

职责：

- 下载结果图片
- 下载 JSON 解析结果
- 导出调试信息（含 `request_id`，便于与日志关联排查）

### 6.2 后端模块

#### 6.2.1 preprocess

职责：图像归一化、增强、校正。

#### 6.2.2 layout

职责：页面分行、小节定位、候选符号框提取。

#### 6.2.3 ocr

职责：

- 使用 OCR 识别数字与调号文本
- 输出候选文本、边界框、置信度

#### 6.2.4 symbols

职责：

- 用 OpenCV 识别点、线、小节线等 OCR 不稳定元素
- 建立几何关系

#### 6.2.5 parser

职责：

- 将 OCR/CV 结果组装为乐谱结构
- 识别全局调号、局部转调、倚音组、休止符

#### 6.2.6 music

职责：

- 音乐语义模型
- 调号换算
- 十二平均律音高解码与重编码
- 等音拼写策略
- 转调逻辑

#### 6.2.7 render

职责：

- 局部擦除
- 字形绘制
- 原位贴回
- 导出 PNG 或 SVG 辅助结果

#### 6.2.8 validate

职责：

- 低置信度标记
- 规则一致性检查
- 端到端回归对比

#### 6.2.9 api

职责：

- 提供同步处理接口 `/api/transpose`
- 统一返回 JSON 响应（成功或错误）
- 处理超时和异常

#### 6.2.10 task_manager

职责：

- 处理单次请求的完整生命周期
- 临时存储中间结果（处理完成后清理）
- 记录处理时间和错误日志

#### 6.2.11 storage

职责：

- 临时存储处理过程中的原图、中间结果、输出图
- 处理完成后自动清理

#### 6.2.12 llm

职责：

- 用 LangChain 封装模型调用
- 定义结构化输出 schema
- 管理 prompt、重试、fallback 和低置信度纠错链路

#### 6.2.13 logging

职责：

- 统一日志格式（时间戳、级别、模块、请求 ID、消息、可选结构化字段）
- 后端：各 pipeline 阶段打点（预处理、OCR、绑定、移调、渲染），记录耗时与关键中间结果
- 前端：用户操作、API 调用、错误信息，支持开发模式控制台输出
- 支持按请求 ID 串联前后端日志，便于端到端调试
- 错误与 warning 单独标记，便于快速定位

### 6.3 前后端接口建议

- `POST /api/transpose`：提交图片和目标调，同步返回结果

请求（支持 `multipart/form-data` 或 JSON base64，建议 multipart 以节省内存）：

```json
{
  "image": "<base64 encoded image>",
  "target_key": "C"
}
```

响应：
```json
{
  "success": true,
  "output_image": "<base64 encoded image>",
  "score_json": { "measures": [...] },
  "warnings": [
    { "type": "low_confidence", "measure": 5, "message": "..." }
  ],
  "processing_time_ms": 8500,
  "request_id": "req_xxx"
}
```

错误响应：
```json
{
  "success": false,
  "error_code": "KEY_NOT_FOUND",
  "error_message": "无法识别调号，请检查图片清晰度",
  "request_id": "req_xxx"
}
```

**暂不提供**：任务队列、轮询接口、历史记录。

设计目标：一张图进，一张图出，JSON 作为调试/验证辅助。

## 7. 核心数据结构

建议内部统一使用 JSON 兼容的 IR。

### 7.1 顶层结构

```json
{
  "score_id": "req_xxx",
  "source_key": {
    "label": "1=G",
    "tonic": "G",
    "mode": "major",
    "bbox": [48, 36, 72, 24],
    "confidence": 0.99
  },
  "target_key": {
    "label": "1=C",
    "tonic": "C",
    "mode": "major"
  },
  "measures": [
    {
      "number": 1,
      "events": [
        {
          "id": "m1e1",
          "type": "note",
          "degree": 5,
          "accidental": "natural",
          "octave_shift": 0,
          "duration_marks": [],
          "grace": false,
          "bbox": [120, 80, 18, 20],
          "confidence": 0.97
        },
        {
          "id": "m1e2",
          "type": "rest",
          "symbol": 0,
          "bbox": [152, 80, 18, 20],
          "confidence": 0.98
        }
      ]
    },
    {
      "number": 5,
      "events": [
        {
          "id": "m5k1",
          "type": "key_change",
          "label": "1=D",
          "tonic": "D",
          "bbox": [420, 210, 68, 22],
          "confidence": 0.96
        }
      ]
    }
  ]
}
```

说明：`score_id` 为本次请求标识，用于调试与日志关联。`measures[].number` 可为非连续值（如多行谱、跳行等）。

### 7.2 事件设计建议

事件类型建议至少包括：

- `note`
- `rest`
- `key_change`
- `barline`
- `grace_group`
- `unknown`

说明：`barline` 仅用于版面切分或可选标注，可不写入 `measures[].events`。

每个事件建议都保留：

- `bbox`
- `confidence`
- `source_tokens`
- `line_index`
- `measure_index`

对 `note` 事件，建议额外保留：

- `decoded_pitch_pc`
- `decoded_pitch_octave`
- `transposed_pitch_pc`
- `transposed_pitch_octave`
- `render_tokens`

这样在调试错误时可以快速回溯到原图区域。

## 8. 关键算法设计

### 8.1 原调识别算法

目标：从整页中识别全局调号和显式局部转调。

实现方式：

1. 对页首和每行左侧区域优先 OCR
2. 用正则匹配调号模式：`1\s*=\s*([A-G](#|b)?)`
3. 将候选按位置和置信度排序
4. 页首最高置信度候选作为全局调号
5. 行中部再次出现的调号候选作为局部转调事件

优点：

- 简单
- 易解释
- 与当前范围匹配

### 8.2 目标调输入校验

允许输入：

- `C D E F G A B`
- 升号调：`F# C# G#`
- 降号调：`Bb Eb Ab Db`

内部统一映射到 pitch class：

```text
C=0, C# / Db=1, D=2, Eb=3, E=4, F=5, F#=6, G=7, Ab=8, A=9, Bb=10, B=11
```

### 8.3 移调算法

设：

- 原调主音 pitch class 为 `pc_source`
- 目标调主音 pitch class 为 `pc_target`

先计算目标 pitch class 差值：

$$
\Delta_{pc} = (pc_{target} - pc_{source}) \bmod 12
$$

再根据移调方向策略选择实际使用的有符号半音间隔 $\Delta$：

$$
\Delta \in \{\Delta_{pc} - 12, \Delta_{pc}, \Delta_{pc} + 12\}
$$

默认建议使用“最接近原音域”的策略，即选择绝对值最小的 $\Delta$。后续可扩展为 `up`、`down`、`nearest` 三种模式。

对每一个非休止符音符，执行：

1. 根据当前调号上下文，将简谱音符解码为绝对音高 $p_{in}$
2. 执行半音平移：

$$
p_{out} = p_{in} + \Delta
$$

3. 在目标调上下文下，将 $p_{out}$ 重新编码为新的 `degree`、`accidental`、`octave_shift`
4. 根据目标调的升降号偏好选择 `#` 或 `b` 的等音写法

处理规则：

1. 全局调号 `source_key` 要改写为 `target_key`
2. 每个显式 `key_change` 的 tonic 也要执行相同的半音平移（保持与全局调号相同的移调间隔）
3. 每个 `note` 事件都必须走“解码 -> 平移 -> 重编码”流程
4. `rest` 事件保持 `0`
5. 低置信度局部转调不自动改写，进入人工复核列表

局部转调处理详细流程：

```
输入：全局调号 K_global，目标调 K_target，移调间隔 Δ

对每个 key_change 事件（局部转调标记）：
    1. 记录原局部调号 K_local_original
    2. 计算新局部调号：K_local_new = K_local_original + Δ（半音平移）
    3. 重写调号文本为 1=K_local_new
    
对 key_change 后的每个 note：
    1. 按原局部调 K_local_original 解码为绝对音高
    2. 应用移调：p_new = p_old + Δ
    3. 按新局部调 K_local_new 重新编码
```

关键点：局部转调后的音符不是移到全局目标调，而是在新的局部调语境下重新编码。

### 8.4 移调示例

#### 示例 A：逐音符 12-TET 转调

- 输入全局调号：`1=G`
- 目标调：`1=C`
- 原旋律：`5 6 1 2 0 5`

处理过程：

- `5` 在 `1=G` 下先解码为绝对音高 `D`
- `6` 解码为 `E`
- `1` 解码为 `G`
- 对这些音高统一施加从 `G` 到 `C` 的移调间隔
- 再在 `1=C` 语境下重新编码为新的简谱音符

说明：

- `0` 为休止符，不变
- 某些音符最终可能仍显示为原来的数字，但这是“重编码结果”，不是直接复用原字符

#### 示例 B：带局部转调的逐音符转调

**场景**：全局原调 `1=D`，第 5 小节局部转调 `1=G`，目标调 `1=C`

**核心原则**：所有调号（全局+局部）按**同一半音间隔**平移，局部调号后的音符在新的局部调语境下重编码。

计算步骤：

1. **计算全局移调间隔**：D → C = -2 半音（或向下移大二度）
2. **重写全局调号**：`1=D` → `1=C`
3. **重写局部调号**：`1=G` 按同一间隔平移：G + (-2) = F → 重写为 `1=F`
4. **处理局部调号后的音符**：
   - 先按**原局部调** `1=G` 解码为绝对音高
   - 应用全局移调间隔 -2 半音
   - 在**新局部调** `1=F` 语境下重编码

输出：

- 页首调号 `1=D` 重写为 `1=C`
- 第 5 小节调号 `1=G` 重写为 `1=F`
- 第 5 小节后的音符按 `1=F` 的相对音高重新编码

**注意**：不是把 `1=G` 直接转成 `1=C`，而是保持调号间的音程关系，统一平移。

### 8.5 原位替换算法

对于所有需要变化的区域，执行：

1. 获取原 token 的 bounding box
2. 扩边 2 到 4 像素，形成 patch
3. 基于周边像素估计背景颜色或纹理
4. 清除旧字符
5. 按同字号、同基线渲染新字符
6. 将 patch 贴回原图

如果新字符宽度大于旧字符宽度：

- 首选在 patch 内轻微缩放
- 仍无法容纳时，允许在局部白边范围扩展
- 再失败则输出一条“需人工确认”的 warning

## 9. OCR / LangChain / LLM 的使用位置

### 9.1 OCR 适合做什么

- 调号文本识别
- 数字识别
- 局部转调文本识别

推荐优先级：

1. PaddleOCR
2. Tesseract 作为备选

原因：

- PaddleOCR 对中文场景和复杂版面适配更好
- 也可直接识别简单数字和符号文本

### 9.2 LangChain 适合做什么

- 统一封装模型供应商接口
- 定义 OCR 纠错和结构解析的 prompt template
- 强制返回结构化结果，而不是自由文本
- 为低置信度场景配置 retry、fallback 和日志
- 将 LLM 输出映射回后端 IR，便于后续渲染和校验

LangChain 在这里是后端编排层，不是业务逻辑本身。真正的谱面规则、移调算法和渲染仍应由后端代码主导。

### 9.3 LLM 适合做什么

- 修复 OCR 输出中的歧义
- 结合上下文判断某个 `#` 是临时升号还是噪点
- 对无法通过纯规则解释的小节给出候选解析

### 9.4 不建议让 LLM 直接做什么

- 直接读取整页并输出完整简谱 JSON
- 直接决定最终渲染位置
- 在无中间结构约束的情况下自由解释整页内容

原因：

- 可复现性差
- 调试困难
- 幻觉成本高

## 12. 开发里程碑

目标：2-3 个月内完成可用的单图移调原型。

### 12.1 第 1 周：需求冻结与样本准备

- 冻结识别范围（仅印刷简谱、仅单张图片）
- 收集 20 页印刷简谱样本（覆盖常见调号、局部转调、倚音）
- 定义简化版 JSON IR（仅需支持 note、rest、key_change）
- 确定单同步接口协议
- 选定渲染字体
- **搭建前后端日志框架**（统一格式、request_id、开发模式开关）

交付物：

- 20 页样本集（含期望输出）
- 简化字段定义
- 单接口 API 草案
- 日志规范与配置

### 12.2 第 2-3 周：预处理与 OCR 基线

- 后端：图像灰度化、二值化、倾斜校正
- 后端：接入 PaddleOCR，识别数字和调号文本
- 后端：实现 `1=X` 调号正则匹配
- 后端：搭建 LangChain 基础封装与结构化输出 schema
- 前端：上传页面 + 目标调选择 + 同步等待结果页
- 联调：打通 `上传 -> 等待 -> 显示结果` 最小链路

交付物：

- 可识别调号的 demo
- 可上传图片并等待结果的前端
- 端到端同步流程验证

### 12.3 第 4-5 周：符号绑定与 12-TET 移调

- 后端：行切分（投影分析）
- 后端：OpenCV 识别八度点、时值线
- 后端：符号绑定（数字+点+线 -> 音符事件）
- 后端：接入 LangChain/LLM 对低置信度绑定做纠错与补全
- 后端：实现十二平均律解码/编码
- 后端：实现全局调号和局部转调的移调逻辑
- 前端：结果对比视图（原图/输出图）

关键验证点：
- 示例 B 场景：1=D 全局 + 1=G 局部 → 1=C 全局 + 1=F 局部

交付物：

- 可输出 JSON 的解析器
- 可正确移调的音高计算模块
- 带对比视图的前端

### 12.4 第 6 周：原位渲染与首版交付

- 后端：局部 patch 擦除与重绘
- 后端：字体渲染与原位贴回
- 后端：输出 PNG 结果图
- 后端：接入 LangChain/LLM 纠错链路
- 前端：下载结果图和 JSON
- 端到端测试与 bug 修复

首版交付标准：
- 支持 1=C/D/E/F/G/A/B 全局调号
- 支持显式局部转调（如行内 1=G）
- 支持 #/b 升降号、八度点、基础时值线
- 单张图片同步处理，5-15 秒返回
- 20 页样本中至少 15 页端到端通过（首版验收）；后续迭代提升至 20 页 golden set 作为最终验收（见第 15 节）
- LangChain/LLM 用于低置信度纠错与结构补全

暂不支持：
- 批量处理
- 复杂装饰音、连音线
- 任务历史

## 13. 主要技术难点与解决方案

### 13.1 OCR 对点和横线不稳定

问题：

- OCR 容易读到数字，但不擅长八度点和时值线

方案：

- 数字和调号由 OCR 识别
- 点、线、小节线交给 OpenCV 规则处理

### 13.2 原位替换时字符宽度变化

问题：

- `1=Eb` 比 `1=C` 更宽
- 某些局部替换可能挤压原版布局

方案：

- 优先局部缩放
- 允许小范围扩边
- 输出 warning，不强行覆盖异常区域

### 13.3 OCR 错把歌词当谱面内容

问题：

- 歌词数字、排版文字可能误入简谱解析

方案：

- 先做行级谱面区域切分
- 只在谱面行内启用数字解析
- 用上下文规则过滤明显不属于谱面的文本

### 13.4 显式转调与普通文本混淆

问题：

- 行内某些说明文字可能被 OCR 误识别成类似调号

方案：

- 必须满足 `1=X` 正则模式
- 必须位于谱面行有效区域
- 必须通过邻近音乐内容校验

### 13.5 LLM 幻觉

问题：

- LLM 可能凭上下文“编造”不存在的音符

方案：

- 只允许 LLM 处理候选列表
- 通过 LangChain 强制结构化输出和字段校验
- 结果必须可落回原始 bbox
- 不能脱离 OCR/CV 中间结果直接写最终谱面

### 13.6 等音重编码与升降号选择

问题：

- 同一个十二平均律音高在简谱里可能存在多种写法，如 `#1` 和 `b2`
- 如果没有统一的等音策略，输出会在不同目标调下出现不稳定写法

方案：

- 为每个目标调定义默认升降号偏好
- 先保证音高正确，再用规则决定最终写法
- 将“等音写法偏好”设计为可配置项，便于后续优化

### 13.7 日志与调试

要求：

- 前后端必须建立完整日志系统，开发阶段可快速定位问题
- 后端：每个 pipeline 阶段打点（preprocess、layout、ocr、symbols、parser、music、render），记录耗时、关键输入输出摘要
- 前端：记录用户操作、请求参数、响应状态、错误信息
- 统一使用 `request_id` 串联单次请求的前后端日志，便于端到端追踪
- 错误与 warning 使用独立日志级别，便于过滤
- 开发模式：后端可输出更详细的中间结果（如 OCR 候选、绑定决策），前端控制台输出完整请求/响应

方案：

- 后端：`structlog` 或 `loguru`，支持 JSON 结构化输出
- 前端：封装 `console.log` 包装，支持按环境开关
- 响应中返回 `request_id`，前端出错时可将该 ID 提供给开发者用于日志检索

## 14. 数据集建议

由于当前方案不训练检测模型，因此“训练数据集”不是主需求，建议改为准备三类语料：

### 14.1 开发样本集

- 20 页高质量印刷简谱（与 12.1 里程碑一致）
- 用于快速迭代 OCR、规则和渲染逻辑

### 14.2 Golden 测试集

- 20 页覆盖不同调号、不同密度、带转调、带倚音、带休止符的样本
- 为每页保存期望 JSON 和期望输出图

### 14.3 错误案例集

- 收集 OCR 易错页
- 单独保存失败原因和人工修正结果

如果未来升级为训练型方案，可参考：

- `DeepScoresV2`：适合作为通用音乐符号预训练参考
- `MUSCIMA++`：适合作为标注方式和版面分析参考

但它们都不是当前 TuneAI 项目的主数据来源。

## 15. 验收标准

建议当前方案至少定义以下验收口径：

1. 全局调号识别准确率 >= 95%
2. 显式转调识别准确率 >= 85%
3. `0-7` 数字识别准确率 >= 95%
4. 逐音符移调后的音高正确率 >= 95%
5. 输出图中需要变化的区域替换成功率 >= 95%
6. **端到端样本通过**：首版验收为 20 页样本中至少 15 页端到端通过；最终验收为至少 20 页 golden set 可端到端稳定通过（与 12.4 首版交付标准一致）

## 16. 最终建议

从工程可行性看，最稳妥的路线不是“做一个能读懂所有简谱的 AI”，而是做一个：

- 面向印刷简谱（仅 PNG/JPG 图片输入，不支持 PDF）
- 以 OCR 为主
- 以 OpenCV 几何规则补足
- 以后端规则为主、以 LangChain 驱动的 LLM 做低置信度纠错
- 内部按十二平均律执行逐音符移调
- 输出阶段重写调号、转调标记和所有发生变化的音符簇
- 输出尽量保留原图布局

这个范围对单人 2 到 3 个月是现实的。

最值得优先打通的最小闭环是：

`图片输入 -> 识别谱面 -> 建立调号上下文 -> 逐音符解码到 12-TET -> 半音移调 -> 重新编码并局部覆盖 -> 输出新图`

只要这条链路稳定，再逐步增强倚音、复杂时值和更多边缘情况即可。
