# TuneAI 项目架构与模块说明

> 基于《TuneAI_完整执行方案》整理，前后端分离架构。

---

## 1. 架构总览

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              用户浏览器 (Browser)                             │
├─────────────────────────────────────────────────────────────────────────────┤
│  前端 (Frontend)                                                             │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │
│  │   upload    │ │loading-state│ │result-viewer│ │download-panel│            │
│  └──────┬──────┘ └──────┬──────┘ └──────┬──────┘ └──────┬──────┘            │
│         │               │               │               │                    │
│         └───────────────┴───────────────┴───────────────┘                    │
│                                    │                                         │
│                          Jinja2 模板 + Vanilla JS + 前端日志                  │
└────────────────────────────────────┼─────────────────────────────────────────┘
                                     │ HTTP/HTTPS
                                     │ POST /api/transpose (multipart)
                                     │ GET / (页面), GET /static/*
                                     ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  后端 (Backend) — FastAPI + Python                                            │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  api 层：/api/transpose 同步接口、请求校验、超时、统一 JSON 响应        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                    │                                         │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  task_manager：单次请求生命周期、临时存储、耗时与错误日志               │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                    │                                         │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐         │
│  │preprocess│→│ layout   │→│ ocr      │→│ symbols  │→│ parser   │         │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘ └──────────┘         │
│                                                      │                      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │                      │
│  │  music   │←│  llm     │  │ validate │←───────────┘                      │
│  └────┬─────┘ └──────────┘ └──────────┘                                    │
│       │                                                                      │
│       ▼                                                                      │
│  ┌──────────┐     storage（临时文件）    logging（结构化日志 + request_id）   │
│  │  render  │                                                               │
│  └──────────┘                                                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

**前后端分离要点：**

- **前端**：仅负责 UI、上传、目标调选择、结果展示、下载、前端日志；不包含 OCR/移调/图像处理。
- **后端**：提供静态页面（模板+静态资源）和 REST API；所有识别、移调、渲染在后端完成。
- **交互**：单张图片同步处理，`POST /api/transpose` 一次请求返回结果图 + JSON + warnings。

**说明：**

- `frontend/` 与 `backend/` 在**目录与职责**上完全分离；前端仅含模板、静态资源与前端 JS 模块，后端仅含 API 与核心流水线。
- 运行时由**后端**提供页面：`run.py` 启动 FastAPI，将 `frontend/templates` 作为 Jinja2 模板目录，`frontend/static` 作为静态目录；前端不单独起一个 Node 服务。
- 若后续改为 SPA（如 React）+ 独立前端构建，只需将 `frontend/` 改为 SPA 工程，后端仅提供 `GET /` 与 `GET /static/*` 或交由 Nginx 托管静态资源，接口仍为 `POST /api/transpose`，架构不变。

## 3. 前端模块与文件对照

| 模块名 | 职责 | 主要文件 |
|--------|------|----------|
| **upload** | 选择/拖拽上传图片、输入目标调、发起 POST /api/transpose | `frontend/static/js/upload.js` |
| **loading-state** | 同步等待时展示「处理中」、失败与错误提示 | `frontend/static/js/loading-state.js` |
| **result-viewer** | 展示原图与结果图、高亮变化区域、warnings/低置信度提示 | `frontend/static/js/result-viewer.js` |
| **download-panel** | 下载结果图、解析 JSON、导出调试信息（含 request_id） | `frontend/static/js/download-panel.js` |
| **logger** | 前端日志：用户操作、请求/响应摘要、错误；控制台与可选上报 | `frontend/static/js/logger.js` |

**页面与入口：**

- `frontend/templates/index.html`：上传页，引入 `upload.js`、`loading-state.js`、`logger.js`。
- `frontend/templates/result.html`：结果页，引入 `result-viewer.js`、`download-panel.js`、`logger.js`。
- `frontend/static/js/app.js`：全局入口（若需简单路由或公共逻辑）。

## 4. 后端模块与文件对照

| 模块名 | 职责 | 主要文件 |
|--------|------|----------|
| **api** | 同步接口 POST /api/transpose、页面路由、统一 JSON、超时与异常 | `backend/tuneai/api/routes.py` |
| **task_manager** | 单次请求生命周期、调用 pipeline、临时结果、耗时与错误日志 | `backend/tuneai/core/task_manager.py` |
| **storage** | 临时存储原图/中间结果/输出图，完成后清理 | `backend/tuneai/core/storage.py` |
| **preprocess** | 灰度、去噪、二值化、倾斜校正、对比度、裁切 | `backend/tuneai/core/preprocess.py` |
| **layout** | 行切分、小节定位、候选符号框 | `backend/tuneai/core/layout.py` |
| **ocr** | PaddleOCR 封装，数字与调号文本，候选+bbox+置信度 | `backend/tuneai/core/ocr.py` |
| **symbols** | OpenCV 八度点、时值线、小节线、几何关系 | `backend/tuneai/core/symbols.py` |
| **parser** | 符号绑定、乐谱结构、输出 score IR（measures/events） | `backend/tuneai/core/parser.py` |
| **music** | 调号换算、12-TET 解码/编码、移调、局部转调、等音策略 | `backend/tuneai/core/music.py` |
| **llm** | LangChain 封装、结构化输出、低置信度纠错与补全 | `backend/tuneai/core/llm.py` |
| **render** | 局部擦除、字形绘制、原位贴回、导出 PNG | `backend/tuneai/core/render.py` |
| **validate** | 低置信度标记、规则一致性、端到端校验 | `backend/tuneai/core/validate.py` |
| **logging** | 结构化日志、request_id、各阶段打点与错误 | `backend/tuneai/logging_config.py` |

**数据与配置：**

- `backend/tuneai/schemas/request_response.py`：API 请求/响应体。
- `backend/tuneai/schemas/score_ir.py`：乐谱 IR（与执行方案 §7 一致）。
- `backend/tuneai/config.py`：超时、临时目录、LLM/OCR 等配置。

## 5. 前后端数据流（分离边界）

```
前端                                    后端
────                                    ────
1. 用户选择图片 + 目标调
2. 前端校验（格式、目标调枚举）
3. POST /api/transpose (multipart: image, target_key)
   ────────────────────────────────────────→
4. 生成 request_id，写入 response
5. 接收图片 → storage 临时保存
6. task_manager 调用 pipeline：
   preprocess → layout → ocr → symbols → parser
   → [llm 纠错] → music(移调) → validate → render
7. 输出：output_image, score_json, warnings
8. 清理临时文件
   ←────────────────────────────────────────
9. 返回 JSON: { success, output_image, score_json, warnings, request_id }
10. 前端：loading 结束，展示 result-viewer
11. 前端：download-panel 提供下载与 request_id 调试
```

- **边界**：所有「识别 → 移调 → 渲染」都在后端完成；前端只发一张图+目标调，收结果图+JSON+warnings+request_id。
- **日志**：后端全链路带 request_id；前端日志可记录同一 request_id，便于前后端联调与问题排查。

## 6. 接口约定（前后端契约）

- **唯一业务接口**：`POST /api/transpose`
  - 请求：`multipart/form-data`，字段 `image`（文件）、`target_key`（如 `C`、`G`、`Eb`）。
  - 成功响应：`success: true`，`output_image`（base64 或 URL 依实现）、`score_json`、`warnings`、`processing_time_ms`、`request_id`。
  - 错误响应：`success: false`，`error_code`、`error_message`、`request_id`。
- **页面与静态资源**：由后端提供 `GET /`（上传页）、结果页（可由前端跳转或后端重定向）、`GET /static/*`。
- 暂不提供：任务队列、轮询、历史记录；前后端仅围绕「单图同步处理」协作。

## 7. 技术栈小结

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端 | Jinja2 模板 + Vanilla JS | 无 React/Vue，由后端渲染页面并托管静态资源 |
| 后端 | FastAPI + Python | 同步处理，无队列、无数据库、无用户系统 |
| OCR | PaddleOCR | 数字、调号、局部转调文本 |
| 图像/几何 | OpenCV + Pillow + NumPy | 预处理、版面、符号几何 |
| LLM 编排 | LangChain | 纠错、低置信度补全、结构化输出 |
| 日志 | structlog / loguru + 前端 logger | 前后端均带 request_id，便于联调 |

以上架构与模块划分与《TuneAI_完整执行方案》一致，并明确**前后端分离**的目录、模块与接口边界，便于实现与后续扩展（如将前端替换为 SPA 而保持 API 不变）。
