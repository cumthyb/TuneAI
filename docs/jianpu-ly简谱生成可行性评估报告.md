# jianpu-ly 简谱生成可行性评估报告（TuneAI）

## 1. 背景与评估目标

TuneAI 当前主路径是“简谱图片 -> 结构化 `ScoreIR` -> 转调 -> 渲染输出图”。  
本报告评估：是否可以在现有架构中引入 `jianpu-ly`，用于生成可下载的简谱文本（`.ly` 或 `jianpu-ly` 输入文本），并给出低风险落地方案。

## 2. 现状梳理（基于仓库实现）

- 主流程位于 `backend/tuneai/core/application/pipeline.py`，第 `[7/7]` 步目前产出的是渲染 PNG。
- 当前 IR 位于 `backend/tuneai/schemas/score_ir.py`，是扁平事件模型（`NoteEvent`/`RestEvent`）。
- OCR 后过滤逻辑位于 `backend/tuneai/core/domain/filter.py`，当前仅保留单字符 `0-7`：
  - `1-7` 转为音符事件
  - `0` 转为休止
  - 默认 `accidental="natural"`、`octave_shift=0`
- API 成功响应位于 `backend/tuneai/schemas/request_response.py`，当前只返回：
  - `output_image`
  - `score_json`
  - `warnings` 等元信息
- 前端下载入口位于 `frontend/src/components/DownloadPanel.tsx`，当前仅支持下载 PNG 和 JSON。

## 3. jianpu-ly 适配可行性结论

结论：**可行，但应区分“受限 PoC”与“高保真输出”两级目标。**

### 3.1 现在就能做的（Go）

- 用当前 `ScoreIR` 导出单声部、按音高序列的 `jianpu-ly` 文本。
- 作为附加产物返回，不影响现有图片输出与协议兼容。
- 通过开关灰度启用，支持快速回退。

### 3.2 现在不适合承诺的（暂缓）

在不改 IR 的情况下，不建议承诺以下高保真能力：

- 精准时值/节拍分组
- 小节线、反复记号、连音、装饰音
- 复杂局部转调与多声部排版

原因是当前 IR 未保留这些语义，且 OCR 过滤阶段已丢失大量符号信息。

## 4. 关键风险与缓解策略

### 4.1 语义损失风险（最高）

`filter.py` 只保留 `0-7` 数字，导致节奏和高级记谱语义在链路早期即丢失。  
这会使 `jianpu-ly` 输出更像“可读音高草稿”，而非完整排版乐谱。

**缓解**：

- 报告与接口文档明确 PoC 适用范围（单声部、简化节奏）。
- 输出中增加 `warnings` 或 `metadata`，提示该导出为“简化表达”。

### 4.2 契约变更风险

若在 `/api/transpose` 直接加新字段，需要同步后端 schema、前端类型与下载 UI。

**缓解**：

- 采用“新增可选字段，不改现有字段语义”的兼容策略。
- 前端仅在字段存在时展示“下载 .ly”按钮。

### 4.3 性能与稳定性风险

文本导出本身开销较小，但若后续接入更复杂格式转换工具，可能增加处理时间。

**缓解**：

- 第一阶段仅做纯文本拼装，避免引入重渲染依赖。
- 监控新增阶段耗时，超阈值时可一键关闭特性开关。

## 5. 分阶段落地建议

### 阶段 1：最小可行 PoC（不影响现网）

- 新增导出模块：从 `ScoreIR` 生成基础 `jianpu-ly` 文本。
- 在 pipeline 末尾并行生成该文本，但默认不对外展示或仅灰度展示。
- 产出对照样本：人工验证可读性与调号一致性。

### 阶段 2：API/前端开关化接入

- 后端响应新增可选字段（例如 `jianpu_ly_text`）。
- 前端 `DownloadPanel` 增加“下载 .ly”按钮（字段存在时显示）。
- 增加端到端用例，保证旧客户端不受影响。

### 阶段 3：高保真能力建设（按需）

- 扩展 `ScoreIR`：时值、小节、连音、结构事件等。
- 逐步提升导出质量，使 `jianpu-ly` 输出从“音高草稿”演进到“可出版排版输入”。

## 6. 验收标准（建议）

- 正确性：导出序列与目标调音高关系一致。
- 兼容性：现有 `output_image` / `score_json` 不受影响。
- 可回退性：配置开关可即时禁用 `jianpu-ly` 导出。
- 可用性：用户可一键下载 `.ly`，且文本可被下游工具正常处理。

## 7. 监控指标（建议）

- `jianpu-ly` 导出耗时（P50/P95）
- 含导出字段请求占比（灰度覆盖率）
- 下载成功率与失败率
- 解析失败样本数（用于反推 IR 扩展优先级）

## 8. 最终建议

在 TuneAI 当前架构下，引入 `jianpu-ly` 作为**附加导出能力**是可行且性价比高的。  
建议先落地“受限 PoC + 可选返回 + 前端下载开关”，通过真实样本验证价值，再决定是否投入 IR 扩展以追求高保真简谱排版。
